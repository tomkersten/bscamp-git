merge

but stick with me

there are 3 components

to a git repository

(basically)

you often see two of them

blobs, trees, commits

blobs

SHA1 hash of file size & contents

same content == same sha1 == same blob

  • (always)

example

(you can follow along at home)

$ mkdir new_project
$ cd new_project
$ mkdir directory_1
$ mkdir directory_2

$ echo 'Content of README' > directory_1/README
$ echo 'Content of README' > directory_2/README

$ git hash-object directory_1/README
d3f48b93b168b70f7bf91e27e6bbf0ef97d52821

$ git hash-object directory_2/README
d3f48b93b168b70f7bf91e27e6bbf0ef97d52821

$ # [note: created git repository w/ a single commit]
$ git cat-file blob d3f48b93
Content of README

blobs

takeaway:

  • sha1 of file content
  • duplicate file contents will have same sha1
  • (that's important)

blobs, trees, commits

trees

think of them as directories

(collectors of sub-nodes)

each commit has one

composed of either blobs or other trees

  • (sha1 is computed from sub-node sha1s & filenames)

in our example

The tree-level sha1 takes the sub-component names into account...
but the blob computation doesn't
...it's only on file size/contents

(i'm not sure about file mode...but you get the point)

tree takeaways

  • basically directories...contents can be other trees or blobs
  • sha1s computed off of sub-components' sha1s/names
  • difference in tree sha1s means they have different content (or files)
  • (that last one is important)

blobs, trees, commits

  • these are easy ;-)

commits are composed of:

  • the (single) top-level tree sha1
  • the name/email of the author
  • the time
  • the 'parent' & 'committer' may play in as well...not sure
  • (it really doesn't matter)

commits

takeaway:

changing any of the components results in a different commit id

(which makes it a different commit to git)

notable items

  • 'new_feature' commits are together in the log (nice)
  • 'new_feature' commits have different sha1s now
  • (that last one is important ;-))

different sha1s are different commits to git

(therein lies the rub with rebasing)

never rebase on shared commits

(ever)

  • (that's what merging is for)

doing so gives the finger

to everyone else on the project

  • (and is a huge pain to clean up)

however

don't just merge because it's easy

merging generally complicates reading the log

  • (which sucks)
  • (if you pay attention to the repository log)
  • (which you do, right?)

my workflow

branch off of master

$ git checkout -b newfeaturebranch

commit, commit, commit

  • (on newfeaturebranch)

frequently switch to master, pull in recent changes

$ git checkout master

$ git pull

rebase off of your up-to-date master branch

$ git checkout newfeaturebranch

$ git rebase master

when feature is complete

repeat the pull/rebase process

merge changes into master

$ git checkout master

$ git merge newfeaturebranch

if you have conflicts...

  • (during the rebase process)

edit conflicting files and run:

$ git add [conflictingfilename]

$ git rebase --continue

  • (this should be pretty easy/rare if you pull often)
